[[架构]]
>分为两层：Server层和存储引擎层
Server层负责建立连接、分析和执行SQL
>- 核心功能模块
>- 内置函数
>- 跨存储引擎的功能
>存储引擎层负责数据的存储和提取

# 第一步: 连接器
如果想使用MySQL，需要先连接上MySQL服务。
在Linux里面开启MySQL
```bash
# 管理员为root
# -p是密码，不要明文写
mysql -h$ip -u$user -p
```

服务的启动需要经过TCP三次握手，因为MySQL是基于TCP协议传输的
如果MySQL服务正常运行，然后连接器就开始验证你的用户名和密码，如果不对，就会给Access denied for user报错

如果用户名跟密码都没有问题，连接器就会获取该用户的权限。该用户之后==在此连接里==进行的所有操作都是基于此时获取的权限进行判断。
所以如果一个用户进行了连接，中途被管理员修改了权限。那么已存在连接的权限是不会改变的，只有新建立的连接才会改变权限。

> **查看MySQL服务被多少个客户端连接了**
> `show processlist`

> **空闲连接不会一直占用**。
> MySQL设置了wait_timeout这个参数（默认8小时），如果超过了就会自动断开连接
> 可以使用`kill connection +id`==手==动断开连接
> 当一个处于空闲状态的客户端被服务端主动断开连接之后，客户端不会立即知道。只有在==发起下一条请求==的时候才会收到Lost connection to MySQL server during query报错

> **MySQL==服务支持的最大连接数==**
> 由max_connections参数控制，当超过最大连接数时，系统会拒绝请求并返回too many connections报错。

## **MySQL也有长连接和短连接的概念**。
 短连接
 建立连接（TCP三次握手）
 执行sql
 断开连接（TCP四次挥手）
 长连接
 建立连接（TCP三次握手）
 执行sql
 执行sql
 执行sql
断开连接（TCP四次挥手）
因此长连接的好处就是节省了建立和断开连接的过程，一般推荐使用长连接。
缺点：使用长连接可能会导致内存占用过多，因为MySQL==执行查询过程中临时==使用内存管理连接对象，但是这些连接对象的资源只有在连接断掉的时候才会释放。如果MySQL开启的长连接==累计==过多，可能会占用内存过大，被系统强制杀掉，就会发生==MySQL服务==异常重启。

> **解决长连接占用内存的问题**
> 有两种解决方式
> 第一，定期断开长连接。
> 第二，重置连接。==MySQL 5.7版本，实现了 `mysql_reset_connection`函数的接口。==当客户端进行一个大的操作后，调用函数来重置连接。整个过程不用经过重连和重新权限校验，但是会把连接恢复到刚刚==创建完的状态==。

总结：
	与客户端进行TCP三次握手建立连接
	检查用户名和密码，==不对报错==
	如果用户名和密码都对，则获取用户的权限。之后位于此连接的操作都是基于此权限==进行逻辑判断==。


# 第二步：查询缓存
连接器的工作完成后，就可以向MySQL服务发送SQL语句了。MySQL服务==收到==SQL语句之后就会解析出SQL语句的==头一个字段==，判断是什么类型的SQL语句。
如果SQL是查询语句（select语句），MySQL服务就会去查询缓存Query Cach查找==缓存数据==，看看之前有没有执行过这一条命令。查询缓存是以key value的形式存储==在内存中的==，key为==SQL查询语句==，value就是==SQL语句查询的结果==
命中，则返回value给客户端；如果没有命中，则往下执行该SQL语句，把执行的结果存在查询缓存中。
## ***鸡肋***
对于更新比较频繁的表，==查询缓存命中率低==。只要一个表有更新操作，则==这个表==的查询缓存就要清空。当缓存了一个查询结果很大的数据之后，就立刻清空了，没有起到实际的作用。
因此在MySQL 8.0之后，**移除了**这里的查询缓存；在MySQL 8.0之前，如果要关闭查询缓存，也可以通过设置`query_cache_type`参数= DEMAND 来解决。
> 注意这里说的是Server层的查询缓存，不是引擎里面的(Innodb存储引擎中的buffer pool)。


# 第三步：解析SQL（解析器）
在正式执行SQL语句之前，MySQL会用解析器解析SQL语句。
## **词法分析和语法分析**
第一步，词法分析：MySQL会根据你输入的==字符串==，解析出===关键字===。token（keyword+非关键字）
第二步，语法分析：根据词法分析的结果，==语法解析器==会检查语法是否正确。如果语法正确的话就构建==SQL语法树==，方便==后面模块获取==
> 语法不对会在这个阶段报错


# 第四步：执行SQL
## prepare阶段（预处理）
- 检查MySQL中的表或字段是否存在（5.7是在这阶段和上阶段之间，8.0才是在prepare）
- 将SELECT * 中的 * 扩展为所有的列
## optimize阶段（优化）
优化器来完成。主要是确定执行方案。
比如说是否需要使用索引，使用哪些索引。
> 确定具体使用的是哪个索引:
> 可以在select之前加一个explain，看key

## execute阶段（执行）
真正执行，执行器和存储引擎交互。
有3种方法：
### 主键索引==查询==
有3种类型：
- 等值查询：访问类型为const,read_ first_record==指向 InnoDB 主键查询接口(**让存储引擎定位符合条件的第一条记录**)==,read_record指向一个永远返回 `-1` 的==函数==（表示没有下一条记录）
- 范围查询:访问类型为const,read_ first_record指向 InnoDB 主键查询接口, read_record ==指向 `InnoDB` 的「取下一条记录」接口==
- 主键in查询：==访问类型为range或者eq_ref,当成多个等值查询。==

> 如果记录是不存在的，就会向执行器上报记录找不到的错误，然后查询结束。如果记录是存在的，就会将记录返回给执行器
> 执行器从存储引擎读到记录后，接着判断记录是否符合查询条件，如果符合则发送给客户端，如果不符合则跳过该记录。
> 执行器查询的过程是一个 while 循环，所以还会再查一次，但是这次因为不是第一次查询了，所以会调用 read_record 函数指针指向的函数

- 全表扫描
  如果没有用到索引那么优化器就会指定查询类型为ALL
1、执行器第一次查询，会调用 read_first_record 函数指针指向的函数，因为优化器选择的访问类型为 all，这个函数指针被指向为 InnoDB 引擎全扫描的接口，让存储引擎读取表中的第一条记录；
2、执行器会判断读到的这条记录是否满足我们的要求。是的话就返回给客户端。不是则跳过。
3、执行器查询的过程是一个 while 循环，所以还会再查一次，会调用 ==read_record== 函数指针指向的函数，因为优化器选择的访问类型为 all，read_record 函数指针指向的还是 InnoDB 引擎全扫描的接口，所以接着向存储引擎层要求继续读刚才那条记录的下一条记录，存储引擎把下一条记录取出后就将其返回给执行器（Server层），执行器继续判断条件，不符合查询条件即跳过该记录，否则发送到客户端；
4、一直重复上述过程，直到存储引擎把表中的所有记录读完，然后向执行器（Server层） 返回了读取完毕的信息；
5、执行器收到存储引擎报告的查询完毕的信息，==退出循环==，停止查询。
> 实际上Server 层每从存储引擎读到一条记录就返回给客户端一条的，但是客户端要等全部查询结束之后才会显示出来
- 索引下推（MySQL 5.6）
索引下推能减少二级索引在查询的回表操作，因为它把server层的一部分事情交给存储引擎层了。
>联合索引在遇到范围查询时会停止匹配。
MySQL 5.6之前遇到联合索引停止匹配的情况，都会在查到满足第一个条件的==二级索引==时回表查询数据再通过server层进行判断。
MySQL 5.6之后，存储引擎定位到二级索引后，先不执行回表操作，而是先判断一下该索引中包含的列的条件是否成立。如果条件不成立，则直接跳过该二级索引。如果成立，则执行回表操作，将完成记录返回给 Server 层。

# 总结
- 连接器：建立连接，管理连接、校验用户身份；
- 查询缓存：查询语句如果命中查询缓存则直接返回，否则继续往下执行。MySQL 8.0 已删除该模块；
- ==解析 SQL==，通过==解析器==对 SQL 查询语句进行词法分析、语法分析，然后构建语法树，方便后续模块读取表名、字段、==语句类型==；
- 执行 SQL：执行 SQL 共有三个阶段：
	- 预处理阶段：检查表或字段是否存在；将 select * 中的 * 符号扩展为表上的所有列。
	- 优化阶段：基于==查询成本==的考虑， 选择==查询成本最小==的执行计划；
	- 执行阶段：根据==执行计划==执行 SQL 查询语句，从存储引擎读取记录，返回给==客户端==；
